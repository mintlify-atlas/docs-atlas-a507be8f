Tokens are defined in JSON5 files in `src/tokens/` and compiled by Style Dictionary into CSS, JSON, Figma exports, and LLM documentation. This guide covers everything you need to author a new token or modify an existing one.

## Mandatory workflow

After every change, run:

```bash validate.bash
npm run lint && npm run test && npm run build
```

Fix all errors before opening a pull request. The build must pass cleanly.

---

## Token file format

Each token is a JSON5 object with a set of standard fields. JSON5 allows comments, trailing commas, and unquoted keys, which makes the files easier to read and maintain.

```json5 token-example.json5
{
  tokenName: {
    $value: '{reference.to.other.token}', // or a literal like '16px'
    $type: 'color',                        // required
    $description: 'Human-readable description of when and why to use this token.',
    $extensions: {
      'org.primer.figma': {
        collection: 'mode',
        scopes: ['bgColor'],
        group: 'semantic',
      },
      'org.primer.overrides': {
        dark: '{base.color.neutral.9}',
        'dark-high-contrast': '{base.color.neutral.10}',
      },
      'org.primer.llm': {
        usage: ['button-background', 'interactive-element'],
        rules: 'Use for primary interactive backgrounds. Pair with fgColor.onEmphasis for text.',
      },
    },
  },
}
```

## Required fields

<AccordionGroup>
  <Accordion title="$value">
    The token's value. This can be:

    - A **reference** to another token using curly-brace syntax: `'{base.color.blue.5}'`
    - A **literal value**: `'16px'`, `'#0969da'`, `'500ms'`
    - A **color object** with explicit color space:

    ```json5 literal-color.json5
    $value: {
      colorSpace: 'hsl',
      components: [137.1, 62.9, 32.7],
      hex: '#1f883d',
    },
    ```

    Always prefer referencing an existing base token over hard-coding a literal value. Literals bypass theming and are harder to maintain.
  </Accordion>
  <Accordion title="$type">
    The W3C design token type. Common values:

    | Type | Used for |
    |---|---|
    | `color` | All color tokens |
    | `dimension` | Spacing, sizing, border radius, border width |
    | `shadow` | Box shadow tokens |
    | `duration` | Animation durations |
    | `cubicBezier` | Easing curves |
    | `fontFamily` | Font family stacks |
    | `fontWeight` | Numeric font weights |
  </Accordion>
</AccordionGroup>

## Optional fields

<AccordionGroup>
  <Accordion title="$description">
    A short, plain-language description of the token's purpose. Add `$description` for any token whose name alone doesn't fully explain when to use it. This text surfaces in generated documentation and IDE tooltips.

    ```json5 description-example.json5
    $description: 'Muted background for secondary content areas and subtle grouping',
    ```
  </Accordion>
  <Accordion title="$extensions: org.primer.figma">
    Controls how the token appears in Figma variables. See the [Figma integration guide](/guides/figma-integration) for full field reference. Include this block for any functional token that designers should be able to apply directly in Figma.

    ```json5 figma-extension.json5
    'org.primer.figma': {
      collection: 'mode',
      scopes: ['bgColor'],
      group: 'semantic',
      codeSyntax: {
        web: 'var(--bgColor-muted) /* utility class: .color-bg-muted */',
      },
    },
    ```
  </Accordion>
  <Accordion title="$extensions: org.primer.overrides">
    Provides alternative values for specific themes. Only include values that differ from the default `$value`. Missing themes inherit from the base theme automatically.

    ```json5 overrides-extension.json5
    'org.primer.overrides': {
      dark: '{base.color.neutral.2}',
      'dark-dimmed': '{base.color.neutral.4}',
      'light-high-contrast': '{base.color.neutral.3}',
      'dark-high-contrast': '{base.color.neutral.2}',
    },
    ```
  </Accordion>
  <Accordion title="$extensions: org.primer.llm">
    Includes the token in the generated `DESIGN_TOKENS_SPEC.md` file, which is used by AI coding assistants and LLM-based tooling. Add this block for tokens that benefit from explicit usage documentation.

    ```json5 llm-extension.json5
    'org.primer.llm': {
      usage: ['page-background', 'main-content', 'card-background'], // max 3 items
      rules: 'Use as the primary background for pages and content areas. Do NOT use for emphasis or highlighting.',
    },
    ```

    Guidelines:
    - `usage`: 1–3 concrete use cases as short strings
    - `rules`: Include `MUST` / `SHOULD` / `Do NOT` guidance when applicable
    - Only add this metadata to tokens that benefit from explicit documentation; not every token needs it
    - To flag a token as one to avoid, add `doNotUse: true` alongside `rules` explaining the preferred alternative
  </Accordion>
</AccordionGroup>

## Token naming rules

Token names must be **camelCase** or **kebab-case** and may only contain letters, numbers, and `-`. This is enforced by `npm run lint:tokens`.

```json5 naming.json5
// Correct
bgColor: { ... }
accent-muted: { ... }
borderRadius2: { ... }

// Wrong — spaces and special characters are not allowed
"bg color": { ... }
"border_radius": { ... }
```

### The `@`-hack for default values

When a token needs both a standalone default and named sub-variants, use `@` as the key for the default. The `@` is stripped from the compiled output and becomes the default value at that path level.

```json5 at-hack.json5
{
  bgColor: {
    accent: {
      '@': {
        // compiles to --bgColor-accent (the default)
        $value: '{base.color.blue.0}',
        $type: 'color',
      },
      muted: {
        // compiles to --bgColor-accent-muted
        $value: '{base.color.blue.0}',
        $type: 'color',
      },
      emphasis: {
        // compiles to --bgColor-accent-emphasis
        $value: '{base.color.blue.5}',
        $type: 'color',
      },
    },
  },
}
```

Without `@`, you cannot have both `--bgColor-accent` and `--bgColor-accent-muted` — one would collide with or shadow the other.

## Where to place tokens

```
src/tokens/
├── base/          # Primitive values — raw colors, raw sizes
│                  # Never referenced directly in product code
├── functional/    # Semantic tokens — bgColor, fgColor, borderColor,
│                  # spacing, typography, motion, shadows
│   ├── color/
│   ├── size/
│   ├── typography/
│   └── ...
└── component/     # Component-scoped tokens
                   # e.g. --control-bgColor-rest
```

<CardGroup cols={3}>
  <Card title="base/" icon="circle">
    Raw scale values. Add here when you need a new primitive (a new gray step, a new brand color). Never use base tokens directly in components.
  </Card>
  <Card title="functional/" icon="tag">
    Semantic tokens with intent. This is where almost all new tokens belong. Organized by category: `color/`, `size/`, `typography/`, `motion/`.
  </Card>
  <Card title="component/" icon="box">
    Tokens scoped to a specific component namespace such as `control`, `button`, or `input`. Only add here when building a component that owns that namespace.
  </Card>
</CardGroup>

## Adding a new token

<Steps>
  <Step title="Find the right file">
    Open the appropriate file in `src/tokens/`. For a new background color, that's `src/tokens/functional/color/bgColor.json5`. For spacing, look in `src/tokens/functional/size/`.
  </Step>
  <Step title="Add the token definition">
    Include `$value` and `$type` at minimum. Add `$description`, `org.primer.figma`, `org.primer.overrides`, and `org.primer.llm` as appropriate.

    ```json5 new-token.json5
    {
      bgColor: {
        myNewToken: {
          $value: '{base.color.blue.0}',
          $type: 'color',
          $description: 'Background for the new feature highlight area.',
          $extensions: {
            'org.primer.figma': {
              collection: 'mode',
              scopes: ['bgColor'],
              group: 'semantic',
            },
            'org.primer.overrides': {
              dark: {
                $value: '{base.color.blue.4}',
                alpha: 0.1,
              },
            },
            'org.primer.llm': {
              usage: ['feature-highlight', 'promo-banner'],
              rules: 'Use for new feature highlight areas. Pair with fgColor.accent for text.',
            },
          },
        },
      },
    }
    ```
  </Step>
  <Step title="Validate and build">
    ```bash validate.bash
    npm run lint && npm run test && npm run build
    ```

    Fix any validation errors before proceeding. Common issues:
    - Token name contains invalid characters
    - `$type` is missing or misspelled
    - A reference like `'{base.color.blue.99}'` points to a token that does not exist
  </Step>
</Steps>

## Build and validate commands reference

```bash commands.bash
# Lint token files (naming, required fields)
npm run lint

# Auto-fix linting issues
npm run lint:fix

# Auto-fix formatting
npm run format:fix

# Run all tests
npm run test

# Full build (CSS, JSON, Figma, LLM docs)
npm run build

# Build CSS and JSON outputs only
npm run build:tokens

# Build Figma-specific output only
npm run build:figma

# Rebuild LLM guidelines (DESIGN_TOKENS_SPEC.md)
npm run build:llm

# Check color contrast ratios
npm run validate:contrast
```

<Warning>
`DESIGN_TOKENS_SPEC.md` is a generated file. Do not edit it directly — your changes will be overwritten on the next `npm run build:llm` run. Edit the `org.primer.llm` metadata in the source token files instead.
</Warning>
